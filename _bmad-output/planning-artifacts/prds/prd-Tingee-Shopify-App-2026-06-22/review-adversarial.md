# Adversarial Review — Tingee Payment App PRD

## Gate verdict

Proceed with fixes — the core flow is sound but eight unaddressed gaps (three of them critical) will cause silent money loss, double-payment fraud, or App Store rejection if shipped as written.

---

## Critical findings

- **CRITICAL** Duplicate webhook / double-payment — No idempotency key or processed-transaction log is specified anywhere. If Tingee retries a webhook (network hiccup, their retry policy) and the first HTTP 200 was never received, the backend will process the same confirmed transaction twice: calling Shopify Order API twice to mark the order Paid, and — worse — if the first call already succeeded but the 200 was dropped, the second call may create a duplicate payment transaction on the Shopify order, inflating the merchant's recorded revenue. *Missing FR:* Add FR-11b: Before updating any order, the backend must check a durable idempotency store (keyed on `tingee_transaction_id`) and short-circuit if the transaction has already been processed. Response must still be HTTP 200 to prevent further retries.

- **CRITICAL** Webhook arrival after QR expiry — FR-10b states the order stays in its current state after 15 minutes and the buyer is told to re-order. But no FR governs what happens if the buyer actually pays after the 15-minute window (e.g., they queued a scheduled transfer, or the bank delayed the confirmation). The webhook will still arrive; FR-12 will match the amount; FR-14 will call Shopify and mark the original order Paid — silently, after the buyer has already been told to create a new order. The buyer will then pay the new order too, giving the merchant two paid orders for one shipment. *Missing FR:* Add FR-10c: Webhook handler must check the order's QR expiry timestamp (stored at order creation). If the webhook arrives after expiry, reject the auto-confirm; add an Order Note flagging "Late payment received after QR expiry — manual review required"; do NOT auto-mark Paid.

- **CRITICAL** Webhook endpoint has no per-shop routing guard — FR-11 validates HMAC-SHA512 using the merchant's Secret Token. But the PRD does not specify how the backend resolves which merchant/shop a webhook belongs to when the endpoint is shared (e.g., `POST /webhooks/tingee`). If Tingee sends webhooks for Shop A to the same URL Shop B uses (misconfiguration, test environment bleed), and the payload coincidentally matches an order in Shop B's store, HMAC validation passes (wrong shop's secret is never checked if the endpoint is global) and the wrong shop gets an order marked Paid. *Missing FR:* Add FR-11c: The webhook URL registered with Tingee must be shop-scoped (e.g., `/webhooks/tingee/{shop_domain}` or carry a per-shop signing secret), OR the payload must include a verifiable shop identifier that is validated against the secret used for HMAC before any order lookup.

## High findings

- **HIGH** Replay attack window on HMAC — The addendum documents `x-request-timestamp` in the request headers but FR-11 only says "validate HMAC-SHA512" with no timestamp-window check. An attacker who intercepts a valid webhook can replay it minutes or hours later and it will pass HMAC validation. The PRD does not specify that the backend must reject requests where `|now - x-request-timestamp| > N seconds`. *Missing FR:* FR-11 must add: reject any webhook where the timestamp is outside ±300 seconds of server time; log as security warning.

- **HIGH** Partial payment then remainder — A buyer on MBBank (explicitly called out in the PRD glossary as allowing amount editing) sends 50% of the order amount first, then 50% later. Each webhook arrives separately; neither matches the exact amount; both are silently dropped into Order Notes. The merchant sees two Notes but no actionable total, and the buyer has no UI feedback at all — they are stranded on an expired QR page. The PRD has no FR to aggregate partial payments or even detect the pattern. This is a known risk of Static QR + exact-amount-match but is nowhere addressed. *Missing FR:* Either (a) add a Non-Goal explicitly accepting this outcome and define what merchants should tell buyers, or (b) add FR-12b to accumulate partial-payment webhooks by order, sum them, and flag the Order Note with running total vs. order total so the merchant can act.

- **HIGH** Shopify Manual Payment Method API constraint — FR-14 states the app calls Shopify Order API with transaction type `capture` or `sale`. For a Manual Payment Method, Shopify requires creating a transaction via `POST /admin/api/{version}/orders/{id}/transactions.json` with `kind: "capture"` against an existing `kind: "authorization"` transaction, OR with `kind: "sale"` directly. However, Shopify only auto-creates the `authorization` transaction if the checkout was completed with an amount > 0 and the gateway was properly registered. The PRD does not verify that the Manual Payment Method registration (FR-4) creates the authorization transaction at checkout completion. If there is no authorization transaction to capture, the `capture` call will fail, and the retry loop (FR-13) will exhaust without ever succeeding. *Missing FR:* FR-4 and FR-14 must explicitly specify: (a) confirm the gateway registration creates a pending authorization on checkout, (b) if not, use `kind: "sale"` and document the tested API path.

- **HIGH** Secret Token rotation leaves orders in-flight broken — FR-5 allows the merchant to update credentials. But the PRD does not address what happens to pending orders that were created under the old credential. The old credential is overwritten; any webhook from Tingee that uses a signing key derived from the old Secret Token will now fail HMAC validation (FR-11) and be silently dropped. Buyers who paid during the rotation window will never get their order confirmed. *Missing FR:* On credential update, store a grace-period window (e.g., 30 minutes) during which both old and new HMAC secrets are accepted; OR require merchant to confirm no open QR windows exist before allowing rotation.

## Medium findings

- **MEDIUM** Polling when backend is down — FR-10 specifies 5-second polling but does not define what the buyer sees if the polling endpoint returns 5xx, is unreachable, or times out. The NFR says "Order Status Extension continues to work when Tingee API is slow" but that covers QR render, not polling. A buyer whose backend is down will see a spinning indicator indefinitely with no explanation, and may assume payment failed and re-order. *Missing FR:* Define max consecutive polling failures (e.g., 3) after which the UI shows a non-alarming status like "Đang kiểm tra thanh toán… Nếu bạn đã chuyển khoản thành công, merchant sẽ xác nhận trong vài phút." Stop auto-polling; offer a manual "Kiểm tra lại" button.

- **MEDIUM** Countdown timer is purely client-side — FR-10b derives expiry from "15 minutes from order placement" but the countdown lives in the browser. If the buyer reloads the page, the timer resets to 15 minutes (or worse, cannot be reconstructed if the extension doesn't fetch a server-side expiry timestamp). A buyer can reload to get an indefinite QR and a reset countdown while the backend correctly rejects late webhooks (if FR-10c is added). Conversely, without FR-10c, a reloaded timer enables indefinite payment windows. *Missing FR:* The Order Status Extension must fetch a server-side `expiry_at` timestamp on load and derive countdown from that absolute time, not from page load time.

- **MEDIUM** Order Note content not standardised — FR-12 says "add Order Note with full information" but does not define the exact fields or format. Different backend implementations will produce inconsistent notes, making it impossible to build merchant tooling or support scripts on top. This also makes SM-2 and SM-3 unmeasurable. *Missing FR:* Specify a canonical Order Note schema with required fields: `tingee_transaction_id`, `received_amount`, `order_amount`, `timestamp_utc`, `mismatch_reason` enum (AMOUNT_MISMATCH | ORDER_NOT_FOUND | DUPLICATE | LATE_PAYMENT).

- **MEDIUM** GDPR webhooks acknowledged but not scoped — NFR lists `customers/data_request`, `customers/redact`, `shop/redact` as handled. The PRD stores Client ID, Secret Token, and order-level payment metadata. Neither the PRD nor the addendum defines what buyer PII (if any) is stored by the app itself. If the webhook handler logs transaction data (amount, timestamp, order ID), those logs may contain PII. No data retention policy or log scrubbing policy is defined. *Missing FR:* Add explicit statement of what data the app persists, for how long, and the deletion logic for each GDPR webhook type.

- **MEDIUM** No Tingee webhook registration FR — The PRD assumes Tingee will send webhooks to the app's endpoint, but there is no FR that specifies how the webhook URL is registered with Tingee. Is it manual (merchant goes to Tingee dashboard and pastes the URL)? Is it automatic via Tingee API on credential save? If it is manual, UJ-1's "no additional action needed" claim (§4.2) is false. If it is automatic via API, there must be an FR for it including error handling. *Missing FR:* Add FR-3b: On successful credential validation, automatically register the shop-scoped webhook URL with Tingee via their API (if supported), or provide the merchant with the exact URL to register manually — and validate that the URL is registered before showing "Kết nối thành công."

## Low / nice-to-have

- **LOW** FR-7 relies on viewport width (< 768px) to hide/show Deeplink, not actual device capability. A desktop browser in a narrow window will show the Deeplink button but clicking it will fail silently (no native app handler). Consider feature-detecting `navigator.userAgent` or attempting to open the deeplink with a fallback timeout rather than relying solely on viewport.

- **LOW** SM-2 ("≥ 95% auto-confirm within 60 seconds") is unmeasurable without instrumentation FRs. No telemetry, logging, or observability requirements are defined. At minimum, the backend needs structured event logs for webhook receipt time, Shopify API call time, and outcome to compute this metric.

- **LOW** FR-8 specifies QR contains "TINGEE {order_number}" as transfer note. Shopify order numbers are sequential integers starting from 1001 by default. This is easily guessable, enabling a low-effort attack: a bad actor can craft a payment with note "TINGEE 1002" and trigger reconciliation for a different order (assuming order lookup is by order number, not by a collision-resistant token). *Missing FR:* Use a non-sequential, unguessable payment reference (e.g., a UUID or HMAC-derived token stored per-order) as the transfer note, not the bare Shopify order number.

- **LOW** No specified behaviour when the merchant has no active Credential but the "Thanh toán qua Tingee QR" payment method is somehow still registered (e.g., Tingee API was down during FR-5 credential deletion). A buyer who selects this method at checkout will reach the Order Status page and see the extension fail to render — no QR, no error message, no fallback. Define a graceful degradation state for the Order Status Extension when the backend returns "no active credential for this shop."

- **LOW** The addendum notes Shopify Payments Partner revenue share (15% after $1M/year) for Phase 2, but does not address whether Phase 1 transaction processing is subject to any Shopify or Tingee fees. If Tingee charges per-transaction fees, the merchant cost model is not documented, which could affect merchant adoption (SM-4).
